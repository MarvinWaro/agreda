Developer Handoff Spec

AGREDA PHASE III — Court Booking System

Laravel 13 · Inertia + React (TS) · shadcn/ui · MySQL

This document is the build spec for the AGREDA III website + booking system. It is written to be read by a developer (or Claude Code) working directly inside the existing Laravel 13 React starter-kit repo. It covers the database schema, Eloquent models & relationships, routes, the page/component breakdown, the booking-availability logic, and the queued owner-notification job. Pair it with the Wireframes and ERD canvases.

0 · Context & conventions

  • One physical court, four sports. Basketball, Volleyball, Futsal, Pickleball — lines & equipment are adjusted per booking. So a slot is busy for all sports when booked for any one.
  • Booking = request, not instant. Visitor picks sport → date → time slot → submits. Status starts pending; the owner confirms or declines from the admin panel.
  • Payment is in person. No online payment gateway. total_price is computed/stored for reference only.
  • Notifications. A new request fires (a) an in-app admin alert and (b) an automatic Facebook message to the owner — via a queued job so the visitor's submit isn't blocked.
  • Guests don't need accounts. Bookings capture guest_name/guest_phone; user_id is nullable and only set for the admin/owner login.

4 · Availability logic (the important bit)

Put this in an AvailabilityService. GET /api/availability?date=&sport_id= returns each slot's state for the booking grid.

  1. Look up operating_hours for that weekday → generate hourly slots (open→close).
  2. If a court_closures row exists for the date → whole day unavailable.
  3. Load bookings for that court + date where status ∈ {pending, confirmed}. Mark overlapping slots booked (confirmed) or pending. Sport-agnostic — one court, so any sport's booking blocks the slot.
  4. Past slots (today, before now) → disabled.
  5. Everything else → free.
⚠ Race condition

On POST /book, re-check the slot inside a DB transaction with a lock before inserting — two people can submit the same slot seconds apart. Add a partial unique constraint or a lockForUpdate() guard.

5 · Owner notification (queued job)

After a booking is stored, dispatch a queued job so the visitor's request returns instantly while the Facebook message sends in the background.

// BookingController@store, after $booking->save()
NotifyOwnerOfBooking::dispatch($booking);

// app/Jobs/NotifyOwnerOfBooking.php  (implements ShouldQueue)
public function handle(FacebookService $fb): void {
    $fb->sendOwnerMessage(
        "New booking request: {$this->booking->sport->name} on "
        . "{$this->booking->booking_date->format('M j')} "
        . "{$this->booking->start_time}. From {$this->booking->guest_name}."
    );
    $this->booking->notifications()->create([
        'channel' => 'facebook', 'status' => 'sent', 'sent_at' => now(),
    ]);
}
  • Facebook delivery: use the Messenger Send API with a Page access token, OR (simpler) the owner opts into a Facebook webhook bot. Easiest fallback for v1: send to a private group / page via the Graph API. Wrap it in FacebookService so the channel can be swapped.
  • Laravel 13 tip: centralise queue routing with Queue::route(NotifyOwnerOfBooking::class, queue: 'notifications') in a service provider. Use #[Tries(3)] + #[Backoff] attributes so a flaky FB API retries.
  • In-app admin alert is immediate — a pending badge on the dashboard from a simple count query; no job needed.
  • Confirm/decline from admin should notify the requester back (SMS/FB/none for v1) and write a notifications row.

4
Booking flow — page, store, confirmation, race-condition guard.

Admin bookings — table, confirm/decline, dashboard counts.

Queued notification job + FacebookService (stub first, real token later).
7
Public pages + CMS — Home/carousel, About, Facilities, Pricing, FAQ, Contact, and their admin editors.
